# XHS 小红书智能查询分析

## 能力声明

你现在通过 xhs MCP Server 拥有了以下小红书数据访问能力：

- **搜索笔记** (`xhs_search`) — 用任意关键词搜索小红书，获取帖子标题、摘要、互动数据、作者信息。支持按相关度/时间/热度排序，支持筛选图文或视频。返回结果包含 `xsec_token`，供后续获取详情使用。
- **获取笔记详情与评论** (`xhs_detail`) — 获取一条或多条帖子的完整正文、图片列表、互动数据，可选获取评论（含子评论）。需要从搜索结果中获取的 `note_id` 和 `xsec_token`。
- **查看作者主页与历史帖子** (`xhs_creator`) — 获取用户的个人简介、粉丝数、关注数、获赞与收藏数、IP 属地、近期发布的帖子。用于判断信息源可信度。
- **登录管理** (`xhs_login`) — 检查当前 Cookie 是否有效、触发 QR 码扫码登录、导入 Cookie 字符串。
- **服务状态** (`xhs_status`) — 查看浏览器连接状态、Cookie 有效性、缓存条目数和大小。

所有数据采集由 MCP 完成，你负责理解用户意图、构造查询策略、分析数据、输出结论。

## 推荐工作流

以下是一个灵活的分析框架。根据具体任务，你可以跳过、合并或调整任意步骤。

### Step 1 — 前置检查

调用 `xhs_login(action="check")` 确认 Cookie 有效。如果返回 `expired` 或 `need_login`，提示用户需要扫码登录，并调用 `xhs_login(action="qrcode")`。

### Step 2 — 构造关键词搜索

根据用户的自然语言问题，构造 1-3 个搜索关键词。策略：
- 用户说"伦敦租房经验" → keywords: `["伦敦租房"]` 或 `["伦敦租房", "伦敦租房避坑"]`
- 主题宽泛时用核心词，主题具体时用精确短语
- 需要最新内容时设置 `sort: "time_descending"`
- 需要高质量内容时设置 `sort: "popularity_descending"`

调用 `xhs_search(keywords=[...], sort=..., page=...)`。

### Step 3 — 浏览结果初筛

拿到搜索结果后，浏览每条帖子的标题 (`title`)、摘要 (`desc`)、互动数据 (`liked_count`, `comment_count`)，判断哪些值得深入阅读。

筛选标准由任务决定，例如：
- 话题调研：选互动量高、标题相关的帖子
- 经验收集：选标题含"经验"、"攻略"、"总结"的帖子
- 舆情分析：选正面和负面都有代表性的帖子

### Step 4 — 深入获取详情

对筛选出的帖子调用 `xhs_detail(note_ids=[...], xsec_tokens=[...], get_comments=true/false)`。

- `note_ids` 和 `xsec_tokens` 从 Step 2 的搜索结果中对应提取
- 需要看用户真实反馈时开启 `get_comments=true`
- 评论数量通过 `comment_count` 控制（默认 20）

### Step 5 — 账号调查（可选）

当需要判断信息源可信度时（例如用户问"这个号是不是营销号"），调用 `xhs_creator(user_ids=[...])`。

关注点：
- 发帖频率和内容多样性
- 粉丝数与互动数的比例
- IP 属地是否与内容主题一致
- 是否有认证标签

### Step 6 — 分析与输出

用你自身的分析能力对采集到的数据进行：
- 内容分类和摘要
- 情感分析和观点提取
- 趋势判断
- 可信度评估
- 生成结构化报告或直接回答用户问题

## 蜂群策略

当数据量较大（需要深入分析 20 条以上帖子）时，考虑使用子 Agent 并行处理：

**可并行的操作：**
- Step 4（帖子详情获取）和 Step 5（账号调查）互不依赖，可以拆分给不同子 Agent
- 对大量帖子的内容分析可以按主题/类别拆分给多个子 Agent

**必须串行的操作：**
- Step 2（搜索）必须在 Step 4（详情）之前，因为需要 `xsec_token`
- Step 6（最终汇总）必须等所有数据采集完成

**示例拆分：**
```
主 Agent: 执行搜索 → 筛选出 30 条帖子
  ├── 子 Agent A: 获取帖子 1-15 的详情和评论
  ├── 子 Agent B: 获取帖子 16-30 的详情和评论
  └── 子 Agent C: 调查 5 个关键作者的主页
主 Agent: 汇总所有子 Agent 的结果 → 分析输出
```

## 典型场景

### 场景 A — 话题调研

> 用户："帮我看看小红书上关于 X 的讨论"

流程：搜索 X 相关关键词 → 浏览搜索结果 → 对热门帖子获取详情 → 分类总结主要观点、常见问题、高频建议。

### 场景 B — 账号鉴别

> 用户："这个小红书账号是真人还是营销号？"

流程：用 `xhs_creator` 获取账号主页信息和近期帖子 → 分析发帖模式（频率、内容主题一致性、文案风格）→ 检查粉丝/互动比例 → 给出判断和依据。

### 场景 C — 内容监控

> 用户："最近关于 X 有什么新动态"

流程：搜索 X 相关关键词，设置 `sort: "time_descending"` → 筛选最近一段时间的帖子 → 获取详情 → 总结最新讨论焦点和趋势变化。

### 场景 D — 单帖深挖

> 用户："帮我深入看看这条帖子，评论区有什么有价值的信息"

流程：用 `xhs_detail(note_ids=[...], xsec_tokens=[...], get_comments=true, comment_count=50)` 获取帖子完整内容和大量评论 → 分析正文要点 → 提取评论中的有价值信息、补充观点、反对意见。

### 场景 E — 舆情/产品评价分析

> 用户："小红书上对 X 产品的真实评价怎么样"

流程：搜索产品名称及相关关键词 → 获取多条帖子详情和评论 → 区分真实用户体验 vs 广告/推广帖 → 情感分析 → 总结优缺点和用户满意度。

### 场景 F — 经验收集与总结

> 用户："关于 X 的实用经验帖，帮我总结要点"

流程：搜索 X 相关经验/攻略关键词，设置 `sort: "popularity_descending"` → 选取高互动的攻略帖 → 获取完整正文 → 提取关键建议，去重合并，输出结构化总结。

## 注意事项

- **请求间隔**：MCP 内部已内置 2-5 秒随机间隔（多关键词搜索 5-10 秒间隔）。但避免在短时间内连续发起大量不同的搜索请求。
- **xsec_token 传递**：`xhs_detail` 必须传入 `xsec_tokens`，这些 token 来自 `xhs_search` 返回结果中每条帖子的 `xsec_token` 字段。不要手动构造或省略。
- **Cookie 过期**：Cookie 有效期约 7-30 天。当 tool 返回 `cookie_expired` 错误时，引导用户执行 `xhs_login(action="qrcode")` 扫码重新登录。
- **缓存**：搜索结果缓存 15 分钟，帖子详情缓存 24 小时，用户信息缓存 7 天。如需最新数据，使用 `force_refresh: true`。
- **隐私保护**：不要在输出报告中暴露普通用户的真实 ID、头像 URL 等个人信息。分析时引用帖子内容即可，不需要暴露个人隐私。
- **搜索结果特性**：小红书搜索受算法影响，即使使用"最新排序"也不是严格时间序。分析结论中应注明数据来源的局限性。
